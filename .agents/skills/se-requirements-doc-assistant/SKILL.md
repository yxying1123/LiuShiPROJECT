# SE Requirements Doc Assistant

将课程项目需求整理为可评审、可测试、可追踪的需求文档。

## 适用模板

优先使用：
- `模版库-供claudeCode调用/提交文件模版/春季/软件需求规格说明书.md`

若为夏季项目，则将需求内容映射到：
- 项目管理文档中的需求与迭代目标
- 项目总结中的“目标达成与问题复盘”

## 输入

- 项目背景、目标用户、业务场景
- 已有原型/页面/接口草图
- 约束（时间、技术、人员）
- 非功能目标（性能、安全、可用性）

## 输出要求

- 需求采用唯一编号（如 `FR-001`, `NFR-001`, `IF-001`）
- 每条需求必须可验证（含验收方式）
- 区分“已确认”与“待确认”
- 明确边界：范围内 / 范围外

## 核心流程

### 1. 需求分层

- 目标层：课程项目要解决什么问题
- 能力层：系统需要具备哪些能力
- 规格层：输入、处理、输出、约束、异常

### 2. 功能需求提炼

每条功能需求包含：
- 触发条件
- 处理规则
- 成功输出
- 异常处理
- 验收标准

### 3. 非功能需求量化

至少覆盖：
- 性能（响应时间、并发、吞吐）
- 安全（认证、授权、数据保护）
- 可用性（操作步骤、学习成本）
- 可维护性（日志、配置、模块边界）

### 4. 接口与数据需求

- 外部接口：用户接口、软件接口、通信接口
- 内部数据：核心实体、字段约束、关系说明
- 数据一致性与错误恢复策略

### 5. 追踪关系

建立追踪表：
- 需求编号 → 设计模块
- 需求编号 → 测试用例
- 需求编号 → 实现证据

## 文档生成策略

- 不删除模板条目，仅填充或标注“不适用”
- 对重点章节优先给出实质性内容，不写空泛描述
- 对无法确认的信息，统一放入“待确认问题”列表

## 质量检查清单

- 是否存在不可测试的主观表述（如“友好”“高效”）
- 是否缺少异常流与边界条件
- 是否给出明确的接口输入输出
- 是否支持后续设计与测试直接使用
