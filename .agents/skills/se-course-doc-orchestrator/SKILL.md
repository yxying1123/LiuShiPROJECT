# SE Course Doc Orchestrator

面向软件工程实践课程的“文档总控技能”，把项目证据（需求、设计、代码、测试、部署记录）转成课程要求的标准文档。

## 目标

- 按阶段推进文档：需求分析 → 架构设计 → 软件实现 → 测试部署
- 自动识别学期模板：春季或夏季
- 产出“可提交初稿 + 待确认项 + 证据追踪表”
- 控制写作风格：客观、可验证、避免空话

## 模板识别规则

优先从工作区中定位以下路径：

1. `模版库-供claudeCode调用/提交文件模版/春季`
2. `模版库-供claudeCode调用/提交文件模版/夏季`

### 春季常见文档

- 软件开发计划书
- 软件需求规格说明书
- 软件概要设计说明书
- 软件详细设计说明书
- 测试报告
- 用户手册

### 夏季常见文档

- 项目管理文档
- 用户使用说明书
- 人员分工权重
- 项目总结
- 迭代互评表

## 阶段映射

- **需求分析阶段**：需求规格说明 / 需求相关章节
- **架构设计阶段**：概要设计、详细设计 / 架构与模块章节
- **软件实现阶段**：开发计划、实现说明、项目管理迭代记录
- **测试部署阶段**：测试报告、用户手册、上线与交付说明

## 工作流程

### 1) 识别上下文

收集：
- 当前学期模板（春季或夏季）
- 当前项目类型与技术栈
- 已有文档与仓库证据（代码、提交、截图、测试输出）

### 2) 制定文档冲刺清单

为每个阶段给出：
- 必写文档/章节
- 可用证据来源
- 缺口与风险
- 当次冲刺目标（一次最多 1~2 份文档）

### 3) 生成文档草稿

要求：
- 不删除模板原有条目
- 对“不适用”条目标注“不适用 + 原因”
- 重点章节写实证据（数据、图、链接、结果）
- 保留“待确认”占位符，避免编造

### 4) 质量门禁检查

逐项检查：
- 章节覆盖完整
- 用词可测试、可验证
- 与代码和测试结果一致
- 风险、约束、边界条件明确

## 输出格式

每次执行输出 3 部分：

1. **本次目标文档草稿**（按模板章节）
2. **待确认问题清单**（最多 10 条，按优先级）
3. **证据追踪表**（章节 → 证据来源 → 置信度）

## 写作约束

- 不虚构接口、性能数据、测试结果
- 不把“计划”写成“已完成事实”
- 对未完成事项使用明确状态：已完成 / 进行中 / 未开始
- 优先使用项目内真实信息（仓库结构、日志、截图、测试命令）

## 触发方式

当用户表达以下意图时触发：

- “帮我按课程模板写文档”
- “现在到需求/设计/实现/测试部署阶段了”
- “我不知道交付材料还缺什么”
- “把仓库内容整理成可提交文档”
